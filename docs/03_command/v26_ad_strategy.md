# 만톡 광고 수익화 전략 문서

**작성일**: 2026-02-01
**버전**: v27 (v26에서 업데이트)
**대상 독자**: 비즈니스 의사결정자, 기획자

## 문서 개요

본 문서는 만톡 AI 사주 챗봇의 광고 기반 수익화 전략을 정리합니다. AI 비용 구조, 토큰 쿼터 시스템, 광고 수익 모델, 유저 시나리오별 손익 분석을 포함하여 **비개발자도 이해할 수 있도록** 작성되었습니다.

## 1. AI 파이프라인 비용 구조

만톡은 3가지 AI 모델을 단계적으로 사용합니다.

### 1.1 비용 상세

| AI 모델 | 용도 | 입력 비용 | 출력 비용 | 실행 빈도 | 프로필당 비용 |
|---------|------|-----------|-----------|-----------|---------------|
| **GPT-5.2** | 사주 기본 분석 (saju_base) | $1.75/1M 토큰 | $14.00/1M 토큰 | 프로필 등록 시 1회 | **~$0.08** |
| **GPT-5.2** | 운세 분석 (월별/연간) | $1.75/1M 토큰 | $14.00/1M 토큰 | 주기적 갱신 (캐싱) | ~$0.019/건 |
| **Gemini 3.0 Flash** | 채팅 대화 생성 | $0.50/1M 토큰 | $3.00/1M 토큰 | 매 메시지 | **~$0.009/회** |

### 1.2 채팅 1회당 비용 계산

```
채팅 1회 = 유저 질문 + AI 응답
- 입력 프롬프트: 약 5,000 토큰 (사주 데이터 + 대화 히스토리)
- 출력 응답: 약 2,000 토큰 (AI 답변)
- 총 토큰: 7,000 토큰

비용 = (5,000 × $0.50/1M) + (2,000 × $3.00/1M)
     = $0.0025 + $0.0060
     = $0.0085 ≈ $0.009/회
```

### 1.3 일일 비용 계산 (GPT-5.2 캐싱 적용)

GPT-5.2는 프로필 등록 시 1회만 실행되고 **영구 캐싱**되므로, 기존 유저의 추가 API 비용은 거의 $0입니다.

```
신규 유저 첫날:
= GPT-5.2 saju_base $0.08 + 채팅 3회 × $0.009
= $0.08 + $0.027 = $0.107

기존 유저 (캐시 적중):
= 채팅 3회 × $0.009 = $0.027
```

## 2. 토큰 쿼터 시스템

만톡은 AI 비용을 통제하기 위해 **토큰 쿼터 시스템**을 운영합니다.

### 2.1 기본 쿼터

| 항목 | 값 | 설명 |
|------|-----|------|
| **daily_quota** | 20,000 토큰 | 모든 유저에게 매일 자동 지급 |
| **채팅 1회 소비** | ~7,000 토큰 | prompt 5K + completion 2K |
| **무료 채팅 횟수** | 20,000 ÷ 7,000 = **~3회/일** | 광고 없이 가능한 채팅 |
| **questionAnswerMaxTokens** | 2,048 | AI 응답 최대 출력 토큰 (한글 4,000~6,000자) |

### 2.2 쿼터 초과 판단 공식

```javascript
is_quota_exceeded = chatting_tokens >= (daily_quota + bonus_tokens + rewarded_tokens_earned)

// 예시
// chatting_tokens: 25,000 (누적 사용량)
// daily_quota: 20,000
// bonus_tokens: 3,000 (impression 2회 × 1,500)
// rewarded_tokens_earned: 35,000 (영상 광고 1회 시청)
// → 25,000 < 58,000 → 쿼터 OK (여유 있음)
```

> **v27 변경**: `bonus_tokens`는 Native impression/click 시 즉시 서버 저장 (이전에는 "대화 재개" 버튼 클릭 시에만 저장 → 유실 발생)

### 2.3 토큰 종류

| 토큰 타입 | 획득 방법 | DB 컬럼 | v27 변경 |
|----------|-----------|---------|----------|
| **daily_quota** | 매일 자동 리셋 | `daily_quota` (고정 20,000) | - |
| **bonus_tokens** | Native 광고 impression/click 보상 | `bonus_tokens` | **즉시 서버 저장** |
| **rewarded_tokens_earned** | 보상형 영상/네이티브 선택 | `rewarded_tokens_earned` | **2버튼 선택** |

### 3.2 광고 노출 로직

```
매 2번의 메시지마다 Native 광고 1개 삽입
예시:
- 메시지 1~2: 일반 채팅
- 메시지 2: 광고 #1 (impression → 1,500 토큰 자동)
- 메시지 3~4: 일반 채팅
- 메시지 4: 광고 #2 (impression → 1,500 토큰 자동)
...
- 무제한 반복 (maxCount=9999)
```

### 3.3 핵심: onAdImpression 강제 보상 로직 (v27 즉시 서버 저장)

**중요**: 유저가 클릭하지 않아도 광고가 화면에 표시되면 자동 보상

```
1. Native 광고가 채팅 스크롤에 나타남 (화면 내 50% 이상)
   ↓
2. Google Mobile Ads가 onAdImpression 콜백 자동 발동
   ↓
3. [v27] 앱이 즉시 2곳에 저장:
   ① 클라이언트 state: bonus_tokens += 1,500
   ② 서버 RPC: add_ad_bonus_tokens(1,500) → DB 즉시 반영
   ↓
4. 유저는 추가 채팅 가능 (1,500 토큰 = ~0.214회)
```

**v27 핵심 개선**:
- 이전(v26): "대화 재개" 버튼 클릭 시에만 RPC → 미클릭 시 bonus_tokens 유실
- 현재(v27): impression 발생 즉시 `_saveBonusToServer()` → 유실 방지

## 4. 광고 유형별 수입

### 4.1 광고 타입별 eCPM

| 광고 타입 | eCPM 범위 | 평균 eCPM | 특징 |
|----------|-----------|-----------|------|
| **Native Medium** | $3~15 | **$5** | 채팅 내 자연스러운 버블 형태 |
| **Rewarded** | $10~50 | **$20** | 토큰 100% 소진 시 2버튼 선택 |
| **Interstitial** | $5~25 | **$10** | 페이지 전환 시 전면 노출 |

### 4.2 eCPM이란?

```
eCPM = Effective Cost Per Mille (1,000회 노출당 수익)

예시: eCPM $5
→ 광고 1,000회 노출 시 $5 수익
→ 광고 1회 노출 시 $0.005 수익
```

### 4.3 광고 1회 수입 계산

```
Native Medium (eCPM $5 가정):
= $5 ÷ 1,000 impression = $0.005/impression

Rewarded (eCPM $20 가정):
= $20 ÷ 1,000 impression = $0.020/impression

Interstitial (eCPM $10 가정):
= $10 ÷ 1,000 impression = $0.010/impression
```

## 6. 광고당 손익분기 분석

### 6.1 현재 설정 (impressionRewardTokens = 1,500)

```
광고 1회 impression 수입 (eCPM $5):
= $0.005

광고 보상 1,500 토큰:
= 1,500 ÷ 7,000 토큰 = 0.214 채팅
= 0.214 × $0.009 = $0.0019 비용

마진:
= $0.005 - $0.0019 = +$0.0031/impression (흑자)
```

### 6.2 클릭 보너스 포함 (v27)

```
광고 1회 impression + click 수입:
= $0.005 (impression) + ~$0.15 (click, CPC 기반)
= ~$0.155

광고 보상 3,000 토큰 (impression 1,500 + click 1,500):
= 3,000 ÷ 7,000 = 0.429 채팅
= 0.429 × $0.009 = $0.0039 비용

마진:
= $0.155 - $0.0039 = +$0.151/click (고수익)
```

### 6.3 손익분기점

```
impression만:
= 보상 3,889 토큰까지 흑자

결론:
- 1,500 토큰: 마진 큼 (최적)
- 3,889 토큰: 손익분기점
- 4,000 토큰 이상: 적자

전략: impressionRewardTokens = 1,500 유지 + clickBonusTokens = 1,500 추가
→ impression은 안정 마진, 클릭은 고수익
```

## 8. 서버 추적 시스템

### 8.1 user_daily_token_usage 테이블

| 컬럼 | 설명 | 업데이트 방법 | v27 변경 |
|------|------|---------------|----------|
| **chatting_tokens** | 누적 채팅 토큰 사용량 | DB Trigger 자동 증가 | - |
| **bonus_tokens** | Native impression/click 보상 토큰 | `add_ad_bonus_tokens` RPC | **즉시 서버 저장** |
| **rewarded_tokens_earned** | 보상형 광고 토큰 (영상/네이티브 선택) | `increment_ad_counter` RPC | **2버튼 선택** |
| **ads_watched** | 광고 시청 횟수 | `add_ad_bonus_tokens` RPC | - |
| **native_impressions** | Native 광고 노출 횟수 | `trackNativeImpression()` → RPC | - |
| **native_clicks** | Native 광고 클릭 횟수 | `trackNativeClick()` → RPC | - |

### 8.2 추적 흐름 (v27)

```
[인라인 광고 흐름]
1. 유저 메시지 전송 (2번째마다)
   ↓
2. Native 광고 삽입 → onAdImpression 콜백
   ↓
3. [v27] 즉시 2곳 저장:
   ① state: bonus_tokens += 1,500
   ② _saveBonusToServer(1,500) → add_ad_bonus_tokens RPC
   ↓
4. (선택) 유저가 광고 클릭 시:
   ① state: bonus_tokens += 1,500
   ② _saveBonusToServer(1,500) → add_ad_bonus_tokens RPC
   ↓
5. 쿼터 재계산: chatting_tokens < (daily_quota + bonus_tokens + rewarded_tokens_earned)

[토큰 소진 흐름]
1. chatting_tokens >= 쿼터 → 토큰 소진 감지
   ↓
2. 2버튼 UI 표시:
   - 🎬 영상 광고 → rewarded_tokens_earned += 35,000
   - 📋 네이티브 광고 → rewarded_tokens_earned += 21,000
   ↓
3. 광고 완료 → 대화 재개
```

### 8.3 RPC 함수

| RPC | 용도 | 파라미터 | v27 변경 |
|-----|------|----------|----------|

## 부록: 용어 해설

| 용어 | 설명 |
|------|------|
| **eCPM** | Effective Cost Per Mille. 광고 1,000회 노출당 수익 |
| **CTR** | Click-Through Rate. 클릭률 = 클릭 수 ÷ impression 수 |
| **DAU** | Daily Active Users. 일일 활성 사용자 수 |
| **토큰** | AI 모델의 입출력 단위. 영어 1단어 ≈ 1.3 토큰, 한글 1글자 ≈ 2~3 토큰 |
| **Impression** | 광고가 화면에 표시된 횟수 (클릭 불필요) |
| **Rewarded Ad** | 보상형 광고. 30초 동영상 시청 시 보상 지급 |
| **Native Ad** | 네이티브 광고. 앱 UI에 자연스럽게 섞인 광고 |
| **Interstitial Ad** | 전면 광고. 페이지 전환 시 전체 화면 노출 |
| **RPC** | Remote Procedure Call. Supabase 서버 함수 호출 |
